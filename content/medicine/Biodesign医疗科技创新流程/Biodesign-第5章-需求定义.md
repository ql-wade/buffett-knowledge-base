# 第5章 需求定义（Need Specification）

## 核心观点

### 好的 Need Statement = 一半解决方案

一个精确定义的需求本身就暗示了解决方向。模糊的需求产生模糊的方案，精确的需求产生精确的方案。

### Need Statement 的标准格式

```
[方法/手段] + [目标人群] + [解决什么问题] + [衡量标准]
```

**示例**：
> "通过无创连续监测方法，在新生儿ICU中，将导管相关血流感染发生率从5.3/1000导管日降低至<1/1000导管日"

### 好与坏的 Need Statement

| 类型 | 示例 | 问题 |
|------|------|------|
| ❌ 太宽泛 | "减少医院感染" | 无法指导发明方向 |
| ❌ 包含方案 | "用紫外线消毒导管接口来减少感染" | 限制了发明空间 |
| ✅ 精确且方案无关 | "通过无创方法将CRBSI降至<1/1000导管日" | 明确目标、开放方案 |

### Need Statement 的 SMART 检验

- **Specific**：具体到某个操作中的某个步骤
- **Measurable**：能用客观数据衡量是否达成
- **Actionable**：技术上存在可行路径
- **Realistic**：目标群体和市场规模合理
- **Time-bound**：能在合理周期内验证

## 操作方法

### Need Statement 迭代流程

1. **V1（粗稿）**：从观察笔记直接提炼，允许模糊
2. **V2（验证稿）**：用 Scouting 收集的数据修正
3. **V3（终稿）**：经 3+ 位临床专家确认，通过 SMART 检验

### 常见错误检查

- [ ] 是否包含了具体技术方案？（是 → 删除方案描述）
- [ ] 是否包含量化指标？（否 → 补充数据）
- [ ] 一个不了解背景的人能理解吗？（否 → 重新措辞）
- [ ] 能引导出多种不同技术方案吗？（否 → 可能太窄了）
